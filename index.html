The idea

One program, not a pile of parts

A traditional Linux desktop is an assembly. A compositor here, a panel there, a window manager, a toolkit, a theme engine, a settings daemon — separate programs with separate models, passing messages and hoping they agree. It works. But anything that spans two of them becomes someone's extension, someone's patch, someone's fork.

Starling is built the way you'd build a large application. The compositor, the window manager, the dock, the menu bar and the panels are one program sharing one scene and one render pass. There is no gap between the parts, because there are no parts.

That single decision is why the list below reads the way it does. None of it is a plugin.

Assembled desktops
  • Pick Wayland or X11 at the login screen
  • Tiling means installing a different desktop
  • Blur and transparency via toolkit CSS
  • Theming stops at the toolkit boundary
  • Panel and compositor negotiate over IPC
Starling
  • Wayland and X11 both native, at once
  • Tiling is a switch in Settings
  • Glass is real optics on the GPU
  • One theme, everything, instantly
  • They're the same program
Built in

What falls out of that

These aren't features layered on top. They're what you get when the whole desktop is one application with one view of the world.

Wayland and X11, both native

Both display servers live inside Starling — a full Wayland compositor and a real X server, side by side. Modern apps and X11-only holdouts run together, composited into the same scene, on the same GPU path.

You never choose a session type at login, and you never find an app you can't run because you picked the wrong one.

Chrome · Slack · VS Code · Qt · GTK · X11 apps

Tiling and floating, both first-class

Floating windows by default. Turn on tiling in Settings and windows arrange themselves master-and-stack, the way a tiling WM does — then turn it off and every window returns to exactly where it was floating.

Not an extension, not a different desktop you have to switch to. One switch, same session.

Settings → Tiling Windows

Real glass, not a CSS trick

The dock and panels don't fake depth with a blur filter. A shader gives each panel actual thickness, derives a surface normal from it, and bends the view ray through that normal with Snell's law — then adds a Fresnel rim and a specular highlight riding the same surface.

It refracts what's behind it because it is doing refraction, on the GPU, every frame.

Physically-based · GPU fragment shader

Rendered natively, end to end

No web view, no HTML, no CSS anywhere in the desktop. Every pixel is drawn by the GPU through a UI framework compiled ahead of time to machine code — with no interpreter or virtual machine in the loop.

It animates like a native app because it is one.

Swift, compiled · straight to DRM/KMS
The desktop

And it's a complete desktop

Spaces you can slide between, Mission Control, multi-monitor that handles being unplugged, and a set of first-party apps that ship with it.

  • Spaces and Mission Control. Slide between virtual desktops, carry a window along as you go, or pull back and see everything at once.
  • Monitors that behave. Plug a display in and it arrives with its own dock, menu bar and spaces. Unplug it and your windows come home rather than vanishing.
  • Apps included. Files, Terminal, Settings, Calculator, Text Editor, Image Viewer, and an App Store that installs from the Ubuntu archive.
  • Crashes stay contained. Every app is its own process sharing GPU buffers with the desktop — zero copies, and one app falling over takes only its own window.
Agents

Where AI agents are tenants, not visitors

Being one application pays off here too: the desktop already knows what every window contains, so an agent doesn't have to guess from a screenshot.

  • Their own space. Agents work in a dedicated AI Space with their own windows — visible when you want to watch, out of the way when you don't.
  • They act by label, not by pixel. An agent clicks Save because it found Save — not because something looked like a button at those coordinates.
  • Permission is explicit. Every capability goes through a broker that denies by default and records what was granted.
  • Even Windows apps. Excel, Word and PowerPoint appear as ordinary windows here, and agents drive them through the real application — formulas and all.
Under the hood

How one program does all of it

Starling is built on Flutter's rendering engine — but the entire framework above it was rewritten in Swift, and the desktop it draws is also the display server.

Install

One package, then pick Starling at login

# Ubuntu 25.10 or 26.04, on an amd64 machine
sudo apt install ./starling-desktop_0.1-1_amd64.deb

# then log out, choose "Starling" from the session menu, and sign in
This is an early preview. Starling runs a real desktop on real hardware today, but it is version 0.1: expect rough edges, missing settings, and apps that still need work. Tested on AMD graphics with Ubuntu 25.10 and 26.04.
